# Finance Tracker 完成范围与迁移设计

日期：2026-07-19

状态：已在 `grill-with-docs` 会话中逐段确认

## 目标

项目完成不是仅结束现有 Phase 1，也不是完整复刻旧版单体 HTML。完成状态是形成可日常使用的**本地记账闭环**：从项目内原始银行与支付平台文件开始，经过导入预览、数据清理、审计、商户复核和分类，最终由同一 SQLite 账本生成可信、可筛选、可下钻的网页版统计。

实施采用数据优先路线：先恢复并超过旧版的数据清理质量，再建设统一报表模块，最后迁移可视化。不得为了尽快显示图表而引入旧版 JSON 兼容事实源。

## 完成标准

### 数据质量

- `legacy/categories.md` 是商户关键字与分类关系的历史基线，不是逐笔交易真值。
- 排除内部转账、换汇和失败交易后，至少 95% 的交易能够通过清理后的商户命中基线规则并获得对应分类。
- 95% 是整个账本的质量目标和项目完成门槛，不是单个导入批次的阻断条件。
- 所有未匹配交易必须显式进入待复核，不得被静默归入一个看似确定的分类。
- 新版与旧版的每项重要数据差异必须能够解释为明确修复或有意改变；未解释差异视为缺陷。
- 真实历史数据验证只输出聚合结果，不显示或提交真实交易、姓名、账户标识或来源文件内容。

### 产品闭环

用户能够：

1. 一键扫描项目内账单目录，找出尚未导入的 PDF/CSV；也可以手动选择其他文件。
2. 在写入前查看完整导入预览，包括规范商户、分类、命中规则、审计发现和复核原因。
3. 确认后以原子事务写入 SQLite 账本。
4. 在专注复核模式中按商户组处理未匹配数据，保留单笔例外，并按每次操作自动保存进度。
5. 在账本质量不足时看到持续警告和继续复核入口。
6. 在质量达到门槛后使用完整的本地网页版统计。

### 报表能力

必须迁移旧版的全部报表能力：

- 全局日期、年份、月份、账户、来源、分类和商户筛选；
- 收入、支出、净额、交易数量和待分类 KPI；
- 收支分类占比；
- 分类汇总与下钻；
- 月度收支、累计净额和年度趋势；
- 年度汇总；
- 非消费资金流汇总与明细；
- 筛选后的交易明细。

不迁移旧版以 `localStorage` 实现的分类规则、标签、固定支出设置或独立 HTML 数据副本。这些状态由新版账本和数据库页面管理。

## 系统边界

### 运行形态

- 应用由本机 HTTP 服务承载，浏览器访问动态网页。
- SQLite 是唯一持久事实来源。
- 旧版自包含 HTML 只作为功能、交互和视觉参考，不再是运行路径。
- 所有图表数据由 Python 服务端从 SQLite 计算；浏览器只提交筛选条件并渲染结果。
- 浏览器使用仓库内置并本地提供的轻量图表库，首选 Chart.js，不依赖 CDN。

### 项目内数据目录

- `data/`：正式 SQLite 账本、私密 `config.json` 和其他持久运行数据。
- `.tmp/`：测试、缓存、浏览器产物和处理中间文件。
- `exports/`：用户主动导出的报告或备份。
- `.superpowers/`：设计伴侣生成的临时草图和状态。
- 上述真实或临时数据必须被 `.gitignore` 保护；仓库只保留虚构配置示例。

原始账单目录由应用只读扫描。扫描不会修改源文件，也不会自动写入账本。

## 数据模型

保留现有来源文件、导入批次、交易、分类、对账和审计数据。新增最小模型：

- **规范商户**：报表和复核使用的统一商户身份；
- **商户别名**：精确名称或经确认的包含关键字，与规范商户关联；
- **方向分类规则**：规范商户在收入或支出方向下对应的分类；
- **复核进度**：最后处理位置、跳过状态和更新时间。

交易继续保留原始商户文本和来源记录，同时可关联规范商户与分类。人工覆盖记录在交易上并写入审计记录，不得被后续规则静默覆盖。

规则优先级从高到低为：

1. 人工覆盖；
2. 精确商户别名；
3. 用户确认过影响范围的包含关键字；
4. 未匹配，保持待分类并进入复核。

普通界面不开放任意正则表达式。

## 深模块及接口

复杂逻辑集中在四个深模块后面，浏览器调用方与测试使用相同接口：

### 导入模块

- `preview(inputs)`：解析来源、调用商户解析与对账、生成完整非持久预览。
- `confirm(proposal)`：在验证预览仍有效后，以原子事务写入账本。

### 商户解析模块

- `resolve(transaction)`：按统一优先级返回规范商户、分类、依据和复核状态。
- `preview_rule(rule)`：返回规则将影响的交易集合和冲突，不产生写入。

### 复核模块

- `next_group()`：返回下一个商户复核组和进度。
- `apply(decision)`：应用单笔覆盖或经确认的长期规则，并写入审计。

### 报表模块

- `build(filters)`：一次返回共享同一口径的 KPI、分类、趋势、非消费资金流和明细数据。

解析器、匹配算法、SQL 和图表数据形状都隐藏在这些模块实现中，不泄漏给调用方。

## 用户流程与交互

### 导入

主路径为：扫描账单目录 → 生成统一预览 → 查看审计与规则依据 → 确认写入。单个文件解析失败时，其他文件仍可展示预览，但批次必须清楚标示不完整状态。任何数据库写入失败都整体回滚。

### 专注复核

默认一次展示一个商户复核组，包括交易样本、笔数、金额范围、建议规范商户和建议分类。用户可以确认整组、建立长期规则或展开单笔例外。

交互以键盘为主：

- `Enter`：接受建议并进入下一组；
- 数字键：选择常用分类；
- `S`：跳过；
- `E`：编辑单笔例外；
- `Ctrl+Z`：撤销上一步。

每次操作立即写入进度。重新打开页面时回到上次未完成位置。鼠标操作必须提供同等能力。

### 长期规则

用户修改交易时可以选择“只改本笔”或“保存为长期商户规则”。长期规则保存前必须展示受影响交易和冲突；确认后应用于现有同商户交易和未来导入。规则变更及受影响交易均保留审计轨迹。

### 总览报表

采用总览优先的一页结构：

1. 账本质量状态和继续复核入口；
2. 全局筛选栏；
3. 收入、支出、净额、待分类和非消费资金流 KPI；
4. 分类占比与月度收支；
5. 累计净额、年度趋势与非消费资金流；
6. 分类汇总和筛选后的交易明细。

高级年度分析通过展开或下钻进入，不建立旧版那样独立持有数据的第二套前端状态。

## 统计口径

- 待分类交易仍计入收入、支出、净额与月度趋势，并在分类统计中单列“待分类”。
- 退款与原支出保留为两条账本记录；报表让退款冲减原支出分类，不把退款计作收入，也不排除整组。
- 内部转账、换汇和投资入金或出金不计入收入、支出和净额，但在“非消费资金流”中单独汇总并提供明细。
- 失败交易不进入普通报表，只进入审计和复核。
- 所有 KPI、图表、汇总和明细必须由同一个报表模块口径生成。

## 失败处理

- 文件级错误隔离展示，不丢失其他文件的预览结果。
- 商户规则冲突、对账歧义和无法确定原分类的退款进入待复核，不自动猜测。
- 长期规则应用前必须显示影响范围，应用后允许撤销。
- 数据库写入使用事务，失败时整体回滚。
- 数据库结构迁移前，在项目内创建受 Git 忽略的时间戳备份。
- 账本质量不足不阻止导入，但持续警告。

## 验证策略

### 自动化验证

- 解析器、商户匹配、规则优先级、人工覆盖、退款、非消费资金流和报表聚合使用脱敏合成夹具测试。
- 导入扫描、文件选择、预览、阻断确认、复核、规则影响预览、报表筛选和图表下钻使用仓库内持久化 Playwright 测试。
- 浏览器测试使用项目 `.tmp/` 内的临时数据库和临时文件，不访问正式账本。
- 每个图表都验证其数据与报表模块响应一致，而不是只做截图检查。

### 私有历史数据验证

- 使用项目内真实来源和旧版历史结果进行本地聚合对照。
- 只报告交易数量、覆盖率、日期范围、按币种总额和未复核数量。
- 不提交或上传真实账单、交易引用、账户标识、姓名、截图、视频、追踪或日志。

### 完成检查

- `pwsh -NoProfile -File .\scripts\doctor.ps1`；

- 聚合隐私安全的真实数据验证；
- `git diff --check`、`git status --short` 和完整 diff 自审；
- 对全部显式验收标准逐项核对。

## 实施阶段

1. **环境与存储基础**：保持规范虚拟环境可重建，确保 bootstrap、测试和应用临时文件使用项目 `.tmp/`，切换项目内运行数据路径，完善 Git 忽略和迁移前备份。
2. **数据质量模型**：建立规范商户、别名、方向分类规则和复核进度；将 `legacy/categories.md` 转为初始规则。
3. **数据质量闭环**：实现账单目录扫描、增强导入预览和专注复核，使真实历史账本商户覆盖率达到至少 95%。
4. **统一报表模块**：先用测试锁定退款、待分类和非消费资金流口径，再扩展筛选、趋势和下钻数据。
5. **可视化迁移**：本地内置 Chart.js，迁移总览、分类、趋势、年度汇总、下钻和交易明细。
6. **完整验收**：完成持久化浏览器回归、文档更新、隐私安全的真实数据聚合验证和完整差异自审。

每个阶段独立验收；数据质量阶段未通过前，不开始正式可视化迁移。

## 明确不做

- 不恢复旧版自包含 HTML 作为产品运行路径；
- 不把旧版 JSON 缓存作为新版事实来源；
- 不迁移旧版 `localStorage` 分类、标签或固定支出实现；
- 不允许规则命中后绕过预览自动写入；
- 不在普通界面开放任意正则表达式；
- 不把真实财务数据加入测试夹具或提交到 Git。
