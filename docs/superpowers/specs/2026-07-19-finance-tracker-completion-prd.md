# Finance Tracker 本地记账闭环 PRD

## Problem Statement

目前的 Phase 1 能导入和展示一部分流水，但数据清理质量与旧版分类基线存在明显差距，分类、人工复核、统计和可视化没有形成同一账本驱动的日常闭环。用户无法可信地把项目内银行与支付平台流水转化为可审计、可持续维护的本地网页版记账结果。

## Solution

将应用完成为本地 HTTP 网页应用，以 SQLite 账本作为唯一事实来源。用户先获得完整导入预览，再确认写入；系统按可解释的商户规则处理分类，将未确定项目送入按商户分组的专注复核；同一服务端报表口径提供筛选、KPI、分类、趋势、非消费资金流和交易明细。

完成门槛是参与统计的历史交易中，至少 95% 通过商户规则基线获得分类；内部转账、换汇和失败交易不计入分母，未匹配交易必须明确进入待复核。旧版仅作为功能与数据质量参考，不再作为运行时事实来源。

## User Stories

1. 作为记账用户，我希望扫描项目内账单目录并识别尚未导入的文件，以便无需手工整理导入清单。
2. 作为记账用户，我希望也能手动选择一个或多个账单文件，以便处理目录外的新文件。
3. 作为记账用户，我希望在任何写入前查看完整导入预览，以便理解每笔交易的来源、规范商户、分类和审计状态。
4. 作为记账用户，我希望预览明确显示命中的商户规则及待复核原因，以便不必猜测系统为何得出分类。
5. 作为记账用户，我希望有阻断性导入问题时不写入任何交易，以便账本不会包含半个批次或不可用数据。
6. 作为记账用户，我希望确认一个有效预览后原子写入全部交易，以便失败时不会留下部分导入结果。
7. 作为记账用户，我希望相同来源文件不会被重复导入，以便保持账本与统计准确。
8. 作为记账用户，我希望每笔交易保留原始商户文本和规范商户，以便日后可以审计和修正规则。
9. 作为记账用户，我希望历史商户关键字基线被转换为可维护的规则，以便先恢复旧版的分类覆盖质量。
10. 作为记账用户，我希望精确商户别名优先于包含关键字，以便减少过宽匹配造成的错误。
11. 作为记账用户，我希望单笔人工覆盖始终优先于长期规则，以便特殊交易不会被后续导入或规则更新改回去。
12. 作为记账用户，我希望将一次确认保存为长期商户规则，以便同时处理已有相同商户交易和未来导入。
13. 作为记账用户，我希望保存长期规则前看到会受影响的交易和冲突，以便不发生静默批量修改。
14. 作为记账用户，我希望待复核交易按商户复核组出现，以便一次决定能解决同类问题。
15. 作为记账用户，我希望在商户复核组内保留单笔例外，以便规则不会强迫不相同的交易采用同一分类。
16. 作为记账用户，我希望复核界面支持键盘优先操作、自动保存和恢复进度，以便高效持续完成历史清理。
17. 作为记账用户，我希望看到整个账本的商户覆盖率和待复核数量，以便知道统计是否达到可信门槛。
18. 作为记账用户，我希望质量不足时仍可导入但会看到持续警告，以便不因单个批次的未知商户而中断工作。
19. 作为记账用户，我希望待分类真实现金流仍出现在收入、支出和趋势中，以便总额不因分类未完成而失真。
20. 作为记账用户，我希望退款冲减其原支出分类而不计作收入，以便支出统计符合实际消费。
21. 作为记账用户，我希望内部转账、换汇和投资资金流从日常收支中排除但仍可单独查看，以便现金移动既不污染消费统计也不被隐藏。
22. 作为记账用户，我希望使用日期、年份、月份、账户、来源、分类和商户筛选报表，以便从不同角度理解账本。
23. 作为记账用户，我希望查看收入、支出、净额、交易数量、待分类和非消费资金流 KPI，以便快速判断财务状态。
24. 作为记账用户，我希望查看分类占比、分类下钻、月度收支、累计净额和年度趋势，以便识别消费结构与变化。
25. 作为记账用户，我希望报表的图表、汇总和交易明细始终使用同一口径，以便数字可以互相核对。
26. 作为记账用户，我希望应用无需外部网络服务即可显示图表，以便个人财务数据保持本地。
27. 作为记账用户，我希望真实历史数据验证只输出聚合指标，以便验证质量时不泄露账户、姓名或交易细节。
28. 作为维护者，我希望数据库迁移前产生受 Git 忽略的项目内备份，以便发生迁移故障时可恢复本地账本。

## Implementation Decisions

- 应用保持本地 HTTP 服务与浏览器界面，SQLite 账本是唯一持久事实来源；旧版自包含 HTML 和 JSON 缓存不参与运行时读取。
- 运行时持久数据、临时数据和用户导出分别置于项目管理且受 Git 忽略的位置；真实账单、数据库、备份和浏览器产物不提交。
- 商户解析模块以人工覆盖、精确别名、已确认包含关键字、待复核的顺序解析交易，并返回结果依据与复核状态。
- 复核模块以商户复核组为默认工作单元，支持接受建议、保存长期规则、单笔例外、跳过、撤销和进度恢复。
- 导入模块提供预览和确认两个显式操作；预览不持久化，确认在验证预览仍有效后原子写入。
- 报表模块一次构建共享同一统计口径的 KPI、分类、趋势、非消费资金流和明细；浏览器只提交筛选条件并渲染响应。
- 退款保留两条账本记录并在报表中冲减原支出分类；无法确认原分类的退款进入待复核。
- 内部转账、换汇和投资资金流不计入收入、支出或净额，但作为非消费资金流提供汇总与明细；失败交易仅保留审计与复核可见性。
- 图表库随应用本地提供，首选轻量 Chart.js，不使用 CDN。
- 规则和人工覆盖的写入必须有审计轨迹；长期规则改变前必须展示影响范围并允许撤销。
- 设计遵守 ADR-0001 至 ADR-0008，尤其是本地账本、项目内运行数据、服务端报表、退款和规则优先级决策。

## Testing Decisions

- 主要业务测试从服务层验证外部行为：导入预览与确认、商户解析、覆盖率、复核、退款、非消费资金流和报表结果；测试使用临时项目内 SQLite 数据库与脱敏合成夹具。
- HTTP 集成测试验证真实请求到服务层的边界，包括多文件预览、确认、规则影响预览、复核决策和筛选。
- 浏览器测试验证可见工作流：文件选择、预览、审计状态、确认阻断、复核保存/恢复、筛选和图表下钻。浏览器不计算分类或金额，断言服务端响应与页面展示一致。
- 所有图表验证其数据与报表模块响应一致，而不只依赖截图断言。
- 私有历史数据仅在本地执行聚合验证，输出交易数量、覆盖率、日期范围、币种总额和未复核数量；不得把原始内容写入日志、夹具、截图、追踪或 Git。
- 每个完成切片都运行环境检查、完整测试、差异空白检查与完整差异自审。

## Out of Scope

- 将旧版自包含 HTML、旧版 JSON 缓存或 localStorage 规则作为新版运行时状态。
- 任意正则表达式规则编辑器。
- 自动绕过预览并写入账本。
- 迁移旧版标签、固定支出或独立前端数据副本。
- 云端同步、多用户协作、外部账户直连或网络依赖的图表服务。
- 提交、上传或展示真实账单、账户标识、个人姓名、交易引用或其他财务敏感数据。

## Further Notes

- 实施按数据优先顺序进行：先完成商户规则模型和 95% 覆盖率闭环，再完成统一报表，最后迁移可视化与浏览器验收。
- 95% 是整个账本质量目标，不是单次导入阻断条件；低于目标时必须持续显示质量警告与复核入口。
- 现有服务层、HTTP 服务和浏览器测试是已确认的验证切面。新行为应优先在这些边界中扩展，而不泄露数据库查询、解析算法或图表数据形状给调用方。
