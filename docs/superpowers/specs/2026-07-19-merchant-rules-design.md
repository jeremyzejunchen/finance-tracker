# Issue #16：规范商户与分层规则解析设计

## 目标

将 `legacy/categories.md` 导入为可审计、可维护的初始长期规则；规则在导入预览、确认写入和历史回填中使用同一解析结果。目标是提升商户覆盖率，而不把未确认的交易静默归类。

## 范围

- 新增规范商户、商户别名和方向分类规则的 SQLite 持久模型。
- 按既定优先级解析新导入与既有交易。
- 从 `legacy/categories.md` 幂等导入初始基线规则，并回填符合条件的现有交易。
- 写入逐笔审计记录，并提供只含聚合值的历史验证。

本 Issue 不包含复核页面、规则影响预览 API 或浏览器交互；它们属于 #18。

## 规则模型

### 规范商户

规范商户是用于报表、复核和分类的统一身份。交易继续保留 `merchant_raw` 与现有清理后的 `merchant`，并新增对规范商户的关联。

### 商户别名

别名关联规范商户，匹配类型限于：

- `exact`：与清理后的商户完全相同；
- `contains`：清理后的商户包含确认关键字。

普通界面不支持任意正则表达式。

### 方向分类规则

规则将规范商户在 `income` 或 `expense` 方向映射到现有 `categories` 表中的分类。非消费资金流和失败交易不参与该解析。

## 解析顺序

对可分类交易应用以下顺序：

1. 人工覆盖：保留既有分类和规范商户结果，不被任何规则改写。
2. 精确商户别名：匹配后写入对应规范商户和方向分类。
3. 包含关键字别名：仅在该优先级唯一命中时写入对应规范商户和方向分类。
4. 未命中或同优先级冲突：保持待分类，提供明确原因并进入后续复核。

若金额方向与规则方向不一致，规则不命中。退款、内部转账、外部换汇、投资和失败交易继续由其既有业务口径处理。

## 基线迁移与回填

首次执行时，解析 `legacy/categories.md` 并创建来源为 `legacy_baseline` 的包含关键字别名及其方向分类规则。迁移必须幂等：同一基线规则不重复创建。

回填只处理未分类且非排除交易。每个成功应用的规则都更新规范商户、分类、分类状态与分类原因 `legacy_baseline_rule`，并在 `audit_log` 留下前后状态和规则来源。人工覆盖永远不被改变。

若规则导入或回填失败，事务整体回滚。规则和分类表的结构迁移沿用项目内备份策略。

## 接口边界

`MerchantResolver.resolve(transaction)` 是唯一的规则解析入口，供：

- 导入预览；
- 确认写入；
- 历史回填；
- 后续规则影响预览。

调用方只接收规范商户、分类、分类状态、分类原因和冲突原因；不依赖 SQL 或匹配算法细节。

## 验证

使用合成夹具在现有服务与数据库测试边界验证：

- 人工覆盖优先；
- 精确别名优先于包含关键字；
- 收入/支出方向必须匹配；
- 同优先级冲突保持待分类；
- 基线导入与回填幂等；
- 回填写入审计记录。

私有历史验证仅输出规则数量、重新分类数、覆盖率和日期范围等聚合数据，绝不输出交易、商户、账户或来源文件内容。
