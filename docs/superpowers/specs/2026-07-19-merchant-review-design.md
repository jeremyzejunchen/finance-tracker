# 商户复核组与长期规则工作流设计

## 目标

把待分类流水组织为可恢复的商户复核队列，使用户能够一次确认同一商户的长期分类规则，并提升导入数据的分类覆盖率。该工作流不改变导入流程，也不重新要求用户处理已经可靠分类的流水。

## 范围与术语

- **复核组**：具有相同规范化商户名称和收支方向的一组交易，可跨 `ME` 与 `WIFE` 账户。
- **待复核交易**：未分类、商户识别失败或规则冲突的交易。
- **不在队列中的交易**：已有分类、已有人工覆盖或明确排除的交易。
- **长期规则**：确认后应用于同一商户与方向的现有非人工覆盖交易及未来导入。
- **单笔覆盖**：只修正一笔交易，永久优先于长期规则。

## 后端与数据流

复核组查询为只读操作，按「规范化商户 + 收支方向」聚合待复核交易，并返回商户、方向、笔数、金额合计、日期范围、涉及账户、有限的样本和建议分类。

确认长期规则前，服务先返回受影响历史交易的数量与范围。用户确认后，单个数据库事务执行以下操作：创建或更新受控匹配形式的长期规则、回填该组中未人工覆盖的历史交易、写入审计记录并保存完成状态。不得开放任意正则表达式。

单笔例外只创建人工覆盖，不创建或修改长期规则。人工覆盖始终优先于规则回填。明确排除项不得被规则或覆盖重新纳入普通分类。

跳过操作只保存复核进度，不改变交易、分类或规则。复核进度保存在本地 SQLite 数据库；重新打开页面后，从下一未完成组恢复。

## 页面与交互

新增独立商户复核页。队列顺序为：商户识别失败、规则冲突、其他未分类；同一优先级按交易数量降序。

每个组展示商户、方向、交易笔数、金额、日期范围、账户、有限样本与分类选择。确认长期规则前必须显示影响笔数和范围，并要求明确确认。用户还可以进入该组明细，为特殊条目创建单笔例外，或跳过当前组。

键盘优先操作为：`Enter` 确认长期规则、`S` 跳过、`E` 打开单笔例外。所有操作同时提供鼠标按钮。

无效分类、陈旧复核组、规则冲突和数据库写入失败时，页面必须保留当前组与用户已选分类，显示可读错误，并且不静默写入。

## 验证

自动化单元、接口和浏览器测试仅使用合成流水与临时数据库。覆盖分组筛选、影响预览、确认后的规则回填、人工覆盖优先、跳过与恢复、错误响应以及浏览器关键路径。

实现完成后，额外使用真实本地数据库进行一次只读、汇总级人工验收，例如复核组数量和覆盖率变化。不得输出交易明细、个人信息或原始账单内容，不得写入 Git。

## 非目标

- 不迁移旧版 localStorage。
- 不改变导入、账户归属或币种处理规则。
- 不将真实流水加入测试、日志、夹具、截图或提交。
- 不在本 Issue 中实现报表或可视化功能。
